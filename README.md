# Лабораторная работа №3: Нагрузочное тестирование чат-системы

## Что тестировала

Проверяла максимальное количество одновременных пользователей в real-time чате. 
Запускала параллельно чаты 1-на-1 (клиент-репетитор), в каждом отправляла по 150 сообщений через HTTP API.

## Как это работает

### Реализация нагрузки
Использовала горутины Go — по одной на каждый чат. Они запускаются параллельно и одновременно бомбят сервер запросами:
```go
for i := 0; i < batchSize; i++ {
    go func() {
        createChatAndSpam(login1, login2, 150, file, batchSize)
    }()
}
```

Горутины очень легкие (~2 КБ памяти против ~1 МБ у обычных потоков), поэтому можно запустить сотни одновременно.

### Мониторинг
- **cAdvisor** собирает метрики с Docker контейнеров (CPU, память)
- **Prometheus** сохраняет эти метрики каждые 5 секунд
- После теста Python-скрипт запрашивает данные из Prometheus и строит графики

Важная деталь: метрика CPU показывает суммарную нагрузку по всем ядрам. Например, 1.2 ядра = 4 ядра по 30% каждое (sum по всем ядрам).

## Три сценария тестирования

### Фаза 1: Поиск точки деградации
Запускала тесты с увеличением нагрузки: 50 → 55 → 60 → ... → 200 чатов.
Искала где система начнёт падать.

**Результаты:**
- CPU: среднее 0.83 ядра, макс 0.92 ядра
- Память: среднее 0.29 ГБ, макс 0.36 ГБ
- Задержка: среднее 22.2 мс, макс 30.0 мс
- **Точка деградации: 75-80 чатов** (дальше начинаются ошибки TCP соединений)

### Фаза 2: Статическая нагрузка
Прогнала 20 итераций на фиксированной нагрузке 60 чатов, чтобы проверить стабильность.

**Результаты:**
- CPU: 0.85 ядра (вариация 17%)
- Память: 0.62 ГБ (вариация 8.7%)
- Задержка: 20.9 мс (вариация 8.8%)

Задержка и память стабильны, CPU прыгает из-за GC в Go и переключения горутин.

### Фаза 3: Анализ диапазона
Те же данные что фаза 1, но смотрела на общие тренды: как растут метрики при увеличении нагрузки.

Метрики растут почти линейно, не экспоненциально → код масштабируется хорошо.

## Основные выводы

### Точка деградации
**75-80 одновременных чатов (150-160 пользователей)**

Узкое место — не производительность сервера, а **TCP connection limit ОС**. 
После ~80 чатов новые соединения отклоняются с ошибкой "connection refused".

При этом сам сервер ещё не нагружен:
- CPU всего 0.8-1.2 ядра
- Задержка стабильна на уровне 20-22 мс
- Память 0.3-0.6 ГБ

То есть код эффективный, просто упираемся в лимиты ОС.

### Почему метрики прыгают
- **Память**: пилообразный график из-за GC в Go (аллокация → очистка)
- **CPU**: планировщик горутин + GC паузы
- **Задержка**: стабильна (вариация 8.8%), это главное

### Безопасная нагрузка
**60 чатов (120 пользователей)** — работает стабильно, латенси 21 мс.

## Как запускать тесты

### Поиск точки деградации
```bash
docker compose -p data_base_project \
  -f docker-compose.yml \
  -f docker-compose.degradation.yml \
  up --build degradation-runner
```

### Статическая нагрузка с мониторингом
```bash
# Сначала поднять мониторинг
docker compose -p data_base_project \
  -f docker-compose.yml -f docker-compose.batch.yml \
  up -d prometheus cadvisor node-exporter

# Потом запустить тест (повторить 20 раз)
BATCH_SIZE=60 POST_TEST_SLEEP=15 \
  docker compose -p data_base_project \
    -f docker-compose.yml -f docker-compose.batch.yml \
    up --build batch-runner metrics-collector
```

### Проверка метрик
```bash
ls metrics/batch_many_user_test/
# Должны появиться: cpu.csv, memory.csv, latency_ms.csv, timestamps.env
```

## Что можно улучшить

1. **Горизонтальное масштабирование**: запустить несколько инстансов backend + балансировщик
2. **Connection pooling**: переиспользовать TCP соединения вместо создания новых
3. **Увеличить ulimit**: временное решение, поднять лимит файловых дескрипторов в ОС

## Важные детали для защиты

### Почему не видно классической деградации?
Она есть! Но проявляется не в коде, а в TCP. Метрики растут линейно, значит код эффективен. 
Просто соединения кончаются раньше чем сервер начинает тормозить.

### Про горутины
Это легковесные потоки в Go:
- Обычный поток: ~1 МБ памяти
- Горутина: ~2 КБ памяти
- Поэтому могу запустить 200 горутин = 400 КБ (vs 200 МБ для потоков)

### Про метрику CPU "1.2 ядра"
Это НЕ 120% одного ядра! Это суммарная загрузка всех ядер:
```
sum(rate(container_cpu_usage_seconds_total[60s]))
```
Если 4 ядра по 30% каждое → 4 × 0.3 = 1.2 ядра total.

### Почему не 4 ядра?
Мои тесты — это HTTP запросы (I/O операции). Горутина отправляет запрос → ждёт ответа → обрабатывает. 
Много времени уходит на ожидание сети, не на вычисления. Поэтому CPU не забит полностью.
